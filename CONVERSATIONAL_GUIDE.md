# ChatSpark Conversational Design & Logic Guide
## 📋 개요
본 문서는 ChatSpark 프로젝트에서 수립된 **'사용자 중심의 지능형 대화 설계'** 원칙과 구현 로직을 정리한 가이드입니다. 타 프로젝트의 유연한 챗봇 설계를 위한 참조 모델로 활용할 수 있습니다.

---

## 🚀 1. 핵심 설계 원칙 (Design Principles)

### 1.1 맥락 우선 (Context-First)
- **원칙**: 첫 질문에 즉시 성함/연락처를 요구하지 않습니다.
- **이유**: 정보 수집 전 충분한 가치를 제공하여 신뢰를 형성해야 합니다.
- **가이드**: 최소 2~3회 이상의 유의미한 정보 교환이 이뤄졌거나, 사용자가 "신청", "상세 상담" 등 명시적 의사를 보일 때만 단계 전환(Trigger)을 발생시킵니다.

### 1.2 스마트 브레이크아웃 (Smart Breakout)
- **원칙**: 정보 수집 중에도 언제든지 자유롭게 질문으로 복귀할 수 있어야 합니다.
- **이유**: 챗봇이 이름만 묻는 '취조실'이 되지 않도록 하여 이탈률을 낮춥니다.
- **가이드**: 이름/연락처 입력 단계에서 입력된 텍스트가 '질문 형태'이거나 '특정 키워드'를 포함하면 즉시 상태를 리셋하고 대화 모드로 전환합니다.

## 🛠️ 2. 주요 구현 로직 (Implementation Logic)

### 2.1 지능형 상태 머신 (State Machine)
`script.js`의 `currentStep` 변수를 통해 흐름을 관리하며, 각 단계마다 사용자 입력의 성격(Inquiry vs Info)을 판별합니다.

```javascript
// Smart Breakout 판별 함수 예시
function checkIfInquiry(text) {
    const keywords = ["다른", "뭐", "어떻게", "문의", "궁금", "말고", "아니"];
    return text.length > 10 || keywords.some(k => text.includes(k)) || text.includes('?');
}
```

### 2.2 시스템 프롬프트 최적화 (System Instruction)
`api/chat.js`의 프롬프트에 대화의 유연성을 명시적으로 지시합니다.

- **지시사항**: "사용자가 이름 입력 요청 중 다른 질문을 하면 즉시 수집을 중단하고 질문에 답변하라."
- **트리거**: 확실하게 상담이 무르익었을 때만 `[INQUIRY_COMPLETE]` 태그를 생성하도록 규칙화.

## 🔊 3. 사용자 경험 고도화 (UX Enhancement)

### 3.1 자연스러운 음성 안내 (TTS)
- **기술**: 브라우저 Web Speech API 활용 (추가 비용 없음).
- **최적화**: 'Natural' 또는 'Google' 프리미엄 음성 엔진을 우선 검색하여 기계음을 최소화.
- **편의성**: 헤더의 토글 버튼을 통해 사용자가 음성 안내 여부를 직접 선택.

### 3.2 투명한 개인정보 동의 (Privacy First)
- **UI**: 버튼형 인터페이스를 통해 '예/아니오'를 명확히 선택하도록 유도.
- **강제성 배제**: "동의하지 않으셔도 계속 이용 가능하다"는 메시지를 명시하여 심리적 장벽 제거.

---

## 🧪 4. 사후 관리 및 테스트 가이드

- **회귀 테스트 (Regression Test)**: 로직 수정 후 프론트엔드와 백엔드 간의 데이터 규격(JSON Key)이 일치하는지 반드시 확인합니다.
- **안전장치 (Safety Checks)**: AI의 답변 거부(Safety filter) 상황에서도 챗봇이 멈추지 않도록 백엔드에서 예외 처리를 수행합니다.
